# UI規約

このプロジェクトでは、設計の「唯一の真実(SSOT)」を維持するため、以下の契約を厳守して実装を行ってください。

## 1. コンポーネント選択の優先順位
AIは実装前に必ず以下の基準でコンポーネントを選択すること：
1. **Styled Components (`Bb`接頭辞)**: プロジェクト標準のデザイン（shadcn/uiスタイル）を適用する場合の第一選択。
2. **Primitives (Unstyled)**: `Bb`コンポーネントでは実現できない特殊なレイアウトや、独自のCSS設計が必要な場合のみ使用。
3. **カスタムHTML**: 上記2つで解決不可能な低レイヤー操作が必要な場合に限り、その理由を明記して使用。

## 2. デザイン・スペーシング契約 (8px Grid)
Tailwind CSSによるスタイリングは、以下の「8pxグリッドシステム」の定数以外を原則禁止する：
- **標準余白**: `p-4` (16px), `m-4` (16px)
- **ゆったり余白**: `p-6` (24px) - カードの内側など
- **セクション間隔**: `space-y-8` (32px)
- **角丸**: `rounded-lg` (標準), `rounded-full` (ボタン・アバター)

## 3. データ連携とバリデーション契約
- **バインディングモデル**: `openapi.yml` から生成されたC#モデル（recordクラス）を直接、またはViewModelにラップして使用すること。
- **バリデーション**: UI側で独自のロジックを書かず、モデルに実装された `IValidatableObject` を `EditForm` / `BbForm` で活用してバックエンドと共通化すること。
- **不変性**: リスト表示には `ImmutableArray` を使用し、意図しない副作用を防止すること。

## 4. インタラクションと通知の標準化
- **成功通知**: 保存等の成功時は必ず `sonner` (`BbToaster`) を使い、 `toast.success("メッセージ")` を呼び出すこと。
- **エラー処理**: API例外時は `toast.error()` でユーザーに通知し、詳細はブラウザコンソールにのみ出力すること。
- **日付操作**: `BbDatePicker` 使用時は、必ず `MinDate` プロパティで過去日付の選択可否を制御すること。

## 5. 生成コードのクリーンアップ要求
コード生成後、以下の項目をセルフチェックすること：
- 不要な `using` 指向が残っていないか。
- アクセシビリティ（`aria-label`など）がコンポーネントの標準機能を損なわず付与されているか。
- マジックナンバーではなく、Tailwindのセマンティックなクラス名が使われているか。
